# V0.2 Roadmap

## 目标

把当前的 MVP 从“可演示主链路”推进为“可长期接入项目并持续使用”的工具。

## 当前版本已经具备

1. 项目扫描
2. 项目级脚手架生成
3. 需求转 tracking 与 case 队列
4. 最小 loop 执行
5. 普通用户入口 `start.sh`
6. AI handoff 文件输出

## V0.2 要解决的问题

### 1. Planner 过于模板化

当前问题：
- 所有需求都生成固定 5-case 队列
- 不区分前端、后端、多 repo、数据库、联调等不同任务形态

目标：
- 按 repo 能力、技术栈、需求复杂度生成不同类型和粒度的 case

### 2. Runner 还缺少完整 git/sync 生命周期

当前问题：
- `run-loop` 能执行命令，但不能管理 pull/commit/push/sync recovery

目标：
- 增加 git 工作流门禁
- 增加 sync recovery 与人工介入升级
- 增加 case history 和结构化 run archive
- 增加 runtime status 层，让用户能实时查看 loop 状态

### 3. AI 接入还只是 handoff，不是 executor

当前问题：
- 目前只能生成 `ai_handoff.md`
- 还没有稳定的执行器接入层

目标：
- 设计通用 executor adapter
- 先不绑死具体平台，但预留 Cursor/Codex/Claude Code 接口

### 4. 多 repo/workspace 还不够强

当前问题：
- scanner 能看多 repo
- planner 和 runner 还没有充分利用多 repo 信息

目标：
- 让 scaffold / plan / runner 明确支持 workspace 级项目矩阵

## 实施优先级

1. Planner 增强
2. Loop Git/Sync 增强
3. Executor Adapter
4. 多 repo 与示例增强

## 完成标准

达到以下条件时，可以认为 v0.2 基本完成：

1. `plan` 不再是固定 5-case 模板
2. `run-loop` 具备 git/sync 基础生命周期
3. handoff 升级为更完整的 AI tool bundle
4. 至少有一个多 repo/workspace 示例
